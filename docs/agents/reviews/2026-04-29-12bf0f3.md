## Findings

### Important

- [Skill logic] workflow-src/commands/review-work.md:50-52 — Step 3 bails
  only when `git log <sha>..HEAD --oneline` is empty; it does not also bail
  when the filtered task-commit list is empty. `resolveScope` will return
  `source: "prior-review"` with an empty `commits` array, and the skill body
  does not tell the agent what to do with that.
  - Why it matters: a range containing only `chore:`/`docs:`/merge commits
    (common during release prep) will cause the agent to write a review
    artifact with an empty "Commits reviewed" list, which is misleading and
    consumes the `reviewed_through:` anchor for future runs.
  - Evidence: `filterTaskCommits` at `cli/src/review-scope.ts:34` strips
    everything non-task, and the skill step 7 unconditionally writes the
    artifact.
  - Root cause: spec ambiguity (skill body underspecified).
  - Suggested direction: add a post-filter check in the skill body: if the
    filtered list is empty after applying the scope resolver, report "N
    commits since `<sha>` but none are task commits — nothing to review" and
    skip artifact creation.

### Minor

- [Doc drift] workflow-src/commands/review-work.md:54 — The skill body
  describes the task-commit regex as `^[a-z]+(\(T\d+\))?: ` with the task
  scope optional, but the canonical regex at `cli/src/review-scope.ts:28` is
  `/^[a-z]+\(T\d+\):\s/` with the scope required.
  - Why it matters: agents who implement the filter inline from the skill
    body will match non-task commits like `chore: materialize phase-2 work
    file` that the helper correctly rejects.
  - Evidence: tests at `test/review-scope.test.ts:24-29` confirm
    `chore: materialize phase-2 work file` returns false.
  - Root cause: implementation defect (skill body description).
  - Suggested direction: rewrite the skill line to match the helper regex,
    or delete the inline regex and tell the agent to call
    `filterTaskCommits` directly.

- [Ergonomics] workflow-src/commands/review-work.md:62-67 — The skill tells
  agents they "may call those helpers directly through a small one-off
  script" but does not show one. Agents without a Bun/Node REPL muscle
  memory may reimplement the logic inline.
  - Why it matters: friction on first use; inconsistency risk.
  - Evidence: n/a — ergonomics.
  - Root cause: spec ambiguity.
  - Suggested direction: optionally add a 3-line `bun -e '...'` example, or
    defer and see whether anyone hits it.
