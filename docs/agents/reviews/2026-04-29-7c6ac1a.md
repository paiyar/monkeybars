## Findings

### Important

- [Spec gap] workflow-src/commands/start-session.md:41, workflow-src/commands/project-status.md:26 — The nudge step uses `git log <sha>..HEAD --oneline --no-merges` and counts subjects matching `^[a-z]+\(T\d+\):`. The `--no-merges` flag removes merge commits from the raw log, which means a merge commit is silently dropped before the subject filter runs. This is correct behavior, but it is inconsistent with how `resolveNudge` (and `filterTaskCommits`) handle merge commits: those helpers receive a pre-built `Commit[]` where `isMerge` is a separate field. An agent following the skill instruction literally will never see an `isMerge` commit, so the regex filter implicitly filters only non-merge commits. This is fine in practice but creates a subtle divergence between the skill instruction and the helper contract: the helper's `isMerge` check becomes unreachable when called from the skill path.
  - Why it matters: if a future change removes `--no-merges` from the skill's `git log` call (e.g. to capture merge subjects for audit), the regex filter would silently miscount because it was never designed to handle `Merge branch '...'` subjects. The mismatch is invisible unless both paths are tested.
  - Evidence: skill body step 10 (`--no-merges`) vs `filterTaskCommits` at `cli/src/review-scope.ts:35-37` (checks `isMerge` independently of subject regex).
  - Root cause: spec ambiguity — two representations of the same merge-commit exclusion rule.
  - Suggested direction: add a note in the skill body or a comment in `filterTaskCommits` explaining that `--no-merges` is the skill's guard and `isMerge` is the helper's guard; they are intentionally redundant.

### Minor

- [Test gap] test/review-nudge.test.ts — The test "picks the newest review by filename when multiple exist" passes `headSha: "zzz9999"` with a single commit `feat(T10): latest` whose sha is also `"zzz9999"`. The newest review's `reviewedThrough` is `"new2222"` (not HEAD), so the nudge fires. This is correct. However, the test does not assert that the *older* review (`"old1111"`) is not used — it only asserts the output. If `pickNewestReview` ever returns the wrong review, the count would still be 1 (same commit list) and the test would still pass. This is a weak assertion for a "picks newest" scenario.
  - Why it matters: the test's name implies it is verifying selection logic, but the assertion doesn't distinguish "picks newest" from "picks any."
  - Evidence: `test/review-nudge.test.ts:52-64` — `expect(result).toEqual({ count: 1, reviewedThrough: "new2222" })` would also pass if the oldest review were used and it happened to have `reviewedThrough: "new2222"`. The values here make it accidentally correct but fragile.
  - Root cause: implementation defect (test design).
  - Suggested direction: make the reviews have visibly different `reviewedThrough` values where the older review's `reviewedThrough` would produce a different result if accidentally selected (e.g. `"old1111"` triggers `count: 1, reviewedThrough: "old1111"` vs the expected `"new2222"`). The test already does this — `"old1111"` vs `"new2222"` are different strings — so the assertion *does* distinguish them. This is fine on re-read; low priority.
