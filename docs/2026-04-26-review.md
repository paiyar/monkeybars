# MonkeyBars Usability And Engineering Audit

> Date: 2026-04-26

## What MonkeyBars is

A repo-local, markdown-driven workflow for agentic coding. Canonical commands
in `workflow-src/commands/` are fed through a small TypeScript generator that
emits three adapters from one source: OpenCode slash commands, Claude Code
skills (with `disable-model-invocation: true`), and a Codex plugin. Project
state lives in `docs/plan.md`, `docs/work/phase-N.md`, and `docs/status.md`;
the `monkeybars check` CLI cross-validates those three files against
`git log`/`git status`. Advisory lifecycle hooks (SessionStart /
UserPromptSubmit / Stop / OpenCode compaction) inject context but never block.
One runtime dep (`commander`), Bun for the entrypoint.

## Strengths

1. **Single source of truth → adapters.** `cli/src/generator.ts` is the
   clearest win. Ten commands × three tools × templates with verbatim-inline
   inclusion — and the edit contract ("edit `workflow-src/` first") is
   enforced by tests that read generated files.
2. **Advisory hooks, not blocking hooks.** Explicit design choice in
   `docs/agent-native-workflow-hooks.md`. Re-running install is idempotent
   (`removeMonkeyBarsHooks` keys off a command substring), and
   unparseable config falls back to "warn and skip only that target's hooks"
   (`install.ts:154-171`).
3. **Deterministic check is small and sharp.** `runCheck` in `check.ts`
   encodes eight invariants (phase-label match, state match,
   current-task-is-first-unchecked, last-commit-in-log, WIP documented).
   Exit codes, `--json`, and severity-tagged findings make it easy to drop
   into CI.
4. **Plan lifecycle is thought through.** `docs/plan.md` is explicitly the
   *active* plan, phase numbers are globally monotonic, old plans archive to
   `docs/archive/plans/YYYY-MM-DD-<scope>.md`.
5. **Brownfield/next-release paths are first-class.** `brainstorm-plan.md`
   has explicit modes (greenfield intake, existing-doc synthesis, brownfield
   synthesis, next-release, guided). Spec and architecture templates force
   Current vs Target sections.
6. **Frontmatter contract is strict and minimal.** `name` + `description`
   required, `include_templates` and `opencode_agent` optional. The parser
   throws loudly on malformed frontmatter.
7. **Test surface is well-chosen.** `check.test.ts` tests invariants,
   `install.test.ts` tests idempotency plus end-to-end hook JSON,
   `workflow_content.test.ts` guards the semantics the generator depends on.

## Weaknesses

1. **Bun-on-PATH is a hard install gate.** `bin/monkeybars` is
   `#!/usr/bin/env bun`, `prepack` runs `bun run generate`, and install hook
   commands shell out to `bun "$CLAUDE_PROJECT_DIR"/.claude/hooks/...`. For an
   npm-installable tool this is a real adoption tax — Claude Code users are
   overwhelmingly on Node.
2. **`docs/status.md` parsing is fragile Markdown-as-DB.**
   `parseBulletFields` in `markdown.ts:10-26` matches a specific
   `- **Key:** value` shape. An agent rewriting the file slightly differently
   silently loses fields.
3. **No generation header / CI drift check.** `plugins/monkeybars/` is
   regenerated wholesale every `bun run generate`, but nothing stops a
   contributor from hand-editing a generated file and having the next
   `generate` silently revert it.
4. **No schema for `docs/plan.md` phases.** `check` validates status↔phase
   pointer hygiene but never reads `docs/plan.md` beyond existence.
5. **Install semantics are destructive in ways the docs underplay.**
   `replaceDirectory` unconditionally `rmSync`s `.opencode/commands/` and
   `.claude/skills/` on every install (`install.ts:87`). Custom skills added
   outside MonkeyBars get wiped.
6. **The `Stop` hook fires on every turn end.** In Claude Code, `Stop` fires
   when the assistant finishes responding — not only at session end. The
   hook emits a handoff/completion nudge on every turn.
7. **Only one runtime command: `check`.** A `monkeybars status` (read-only,
   prints active phase/task/next) would replace half of what the
   `start-session` skill does via LLM parsing. A `monkeybars advance` would
   remove regen-introduced drift.
8. **No preflight runner.** `complete-task` asks the agent to read `AGENTS.md`,
   find the preflight block, and execute it — trivially mis-parseable.
9. **`extractPreflightCommands` is dead code.** Exported from `markdown.ts`
   but nothing in `cli/src/` or `test/` calls it.
10. **Codex `plugins/monkeybars` collision risk.** Installs to
    `<project>/plugins/monkeybars/` — clashes with any project using a
    `plugins/` directory for its own plugins. `.codex/plugins/monkeybars/`
    would be conventional.
11. **OpenCode hook only fires on compaction.** Claude/Codex get three
    lifecycle events; OpenCode gets only `experimental.session.compacting`.
12. **Frontmatter parser is hand-rolled.** `parseFrontmatter` in
    `generator.ts:61-95` splits on `:` — values with `:` in them would break.
13. **README install flow doesn't bridge to "now open your coding tool."**
    New users stare at their terminal wondering what slash command means.

## Comparison With Tools In The Same Space

- **BMAD (`bmadcode/BMAD-METHOD`)** — closest analog. Larger,
  agent-persona-driven, web bundles plus IDE packs. MonkeyBars is leaner and
  more opinionated about one state model (plan→phase→task→commit); BMAD has
  no deterministic verifier equivalent to `monkeybars check`.
- **agent-os** — similar target: structured specs, phased tasks, standards
  docs. More polished onboarding. Weaker story on brownfield and plan
  lifecycle.
- **SpecStory / spec-kit** — spec-driven, focused on the spec artifact rather
  than the phase loop. Complementary; MonkeyBars' `spec.md` template is
  thinner.
- **Aider's `CONVENTIONS.md` + repo map** — deep repo-map automation.
  MonkeyBars has no equivalent to repo-map inclusion.
- **Cursor Rules / CLAUDE.md / AGENTS.md baseline** — MonkeyBars' value over
  this is entirely about externalizing plan/phase/task state so context
  compaction doesn't destroy it.
- **Plan-mode features in Claude Code / OpenCode** — in-session. MonkeyBars'
  advantage is that the plan survives the session; weakness is that it
  doesn't bridge to the native plan-mode.
- **Taskfile / Just / Make** — MonkeyBars mentions Taskfile as optional but
  doesn't generate a preflight runner itself.

## Gaps Worth Filling, Ranked By Effort × Impact

- **High impact / low effort:** generation header on every emitted file;
  wire `extractPreflightCommands` into a `monkeybars preflight` subcommand;
  add `monkeybars status`; stop wiping non-MonkeyBars skills on install;
  add a Node-compatible build so Bun isn't a hard install gate.
- **High impact / medium effort:** make `check` verify the active phase
  exists in `docs/plan.md` and that no two unarchived plan phases share a
  number; add `monkeybars advance` for deterministic post-commit state
  transitions; replace markdown-as-DB in `status.md` with YAML frontmatter
  or a structured block.
- **Medium impact / low effort:** refine Claude `Stop` hook matcher;
  document "Bun required" prominently; namespace Codex install under
  `.codex/plugins/monkeybars/`.
- **Future:** `monkeybars doctor` that diffs `workflow-src/` against
  `plugins/monkeybars/` with a CI recipe; OpenCode session-start parity;
  optional per-task timestamp log.

## Bottom Line

MonkeyBars is opinionated, small, and gets the *shape* right — single-source
generator, advisory hooks, deterministic verifier, externalized plan state
with archive semantics. Those four choices are exactly the ones BMAD /
agent-os get too casual about. The gaps are mostly about closing the loop
between the workflow files and the CLI: right now the CLI verifies state the
agent produced, but doesn't help produce it. The biggest single liability
is the Bun runtime gate combined with destructive install semantics — both
are fixable in a day, and both hit first-time users hardest.
